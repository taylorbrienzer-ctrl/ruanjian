# App 成品、Figma 设计与 uni-app 开发约束

## 一、目标调整

当前交付从“网页演示”升级为“移动 App 产品设计 + uni-app 开发交接”。原有网页可以作为视觉参考和内容资产，但最终产品应按手机 App 的使用方式重新组织，而不是把网页缩到手机屏幕里。

## 二、前端界面约束

App 界面中不得出现以下内部说明性词汇：

1. 赛题、评分、得分、A5、软件杯。
2. “赛题能力覆盖”“文档交付说明”等面向参赛过程的表达。
3. 将技术验收点直接写成界面模块标题。

这些内容只保留在文档、答辩材料和接口设计里。用户界面必须使用游客和景区管理方能理解的产品语言。

## 三、App 信息架构

游客端需要形成 5 个一级页面：

1. 首页导览：AI 数字人、语音/文字问答、当前导览状态、快捷问题。
2. 路线推荐：按兴趣、时间、体力、人流推荐游览线路。
3. 景点讲解：灵山大佛、九龙灌浴、灵山梵宫、祥符禅寺等景点知识卡。
4. 行程服务：定位状态、演出提醒、拥挤度、休息点、反馈。
5. 我的：游客偏好、收藏路线、历史问答、语言/声音设置。

管理端作为 App 内运营后台入口或平板/大屏扩展，保留：

1. 今日服务数据。
2. 热门问答。
3. 知识库更新。
4. 数字人形象配置。
5. 游客反馈和服务建议。

## 四、Figma 设计约束

Figma 设计稿应按移动 App 规范建立：

1. 画板尺寸优先使用 390 x 844，兼容 iPhone 15/Android 主流机型。
2. 建立 Design Tokens：颜色、字号、间距、圆角、阴影、状态色。
3. 建立组件库：顶部栏、底部 Tab、数字人舞台、聊天气泡、路线卡、景点卡、数据卡、按钮、标签、输入栏。
4. 首屏不做网页式大标题，不做营销 Hero；首屏直接进入导览工作台。
5. 保留灵山山水、湖青、墨绿、金石、米白、朱砂点缀的视觉系统。
6. 动效标注包括：语音波形、数字人口型状态、云雾背景缓动、路线生成加载。

## 五、uni-app 开发约束

uni-app 项目应按 Vue 单文件组件组织：

1. `pages.json` 定义 5 个游客端页面和底部 Tab。
2. 页面使用 `view`、`text`、`image`、`scroll-view`、`button` 等跨端组件。
3. CSS 使用 rpx/rem 友好尺寸，不依赖 Web 专有布局。
4. 静态数据先本地 mock，后续替换为 `docs/03_api_design.md` 中的接口。
5. 不在 App 界面暴露参赛说明，相关能力通过真实功能呈现。
